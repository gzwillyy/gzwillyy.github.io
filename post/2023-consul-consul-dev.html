Consul 注册中心 docker开发环境部署

发表于 2023-07-25 分类于 Micro

教程地址： https://developer.hashicorp.com/consul/tutorials/day-0/docker-container-agents

在本教程中使用 Consul v1.16.1

在本教程中，您将学习如何部署两个联合的 Consul agents，每个 agents 在单独的 Docker 容器中运行。您还将注册服务并 执行 基本维护 操作。两个 Consul agents将形成一个 小型数据中心 。

通过学习本教程，您将学习如何：

  1. 获取 Consul 的 Docker 镜像
  2. 配置并运行 Consul server
  3. 配置并运行 Consul client
  4. 与 Consul agents 交互
  5. 执行维护操作（备份 Consul 数据、停止 Consul agents等）

本教程以 Docker 为重点，但您将学习的原理也适用于其他容器运行时。

安全警告: 本教程不适合生产用途

有关 Consul 最佳实践，请参阅 [Consul 参考架构](/consul/tutorials/product-deploy/reference-architecture)

有关 Docker 最佳实践，请参阅 [Docker 文档](https://docs.docker.com/)

要扩展您将在本教程中学习的概念，请查看使用 Docker Compose 部署安全的本地 Consul 数据中心教程，以了解有关部署安全 Consul 数据中心的更多信息。

Docker

对于本教程，您需要在计算机上本地安装运行的 Docker。您可以在此处找到在您的特定操作系统上安装 Docker 的说明。

Consul（可选）

如果您想使用本地安装的 Consul 与 容器化 Consul agents 进行交互，请按照此处的说明进行操作，并将二进制文件安装在您的 PATH 中的某个位置。

获取 Docker image

首先，拉取最新的镜像。您将在本教程中使用 Consul 的官方 Docker 镜像。

通过列出匹配的 Docker 镜像来检查镜像是否已下载consul。

docker images -f 'reference=consul'
REPOSITORY     TAG      IMAGE ID        CREATED             SIZE
consul       latest    8b6c5f52aa82   18 hours ago   149MB

配置并运行 Consul Server

接下来，您将使用 Docker 命令行标志将 agents 作为服务器启动、配置网络并在一台服务器启动时引导数据中心。

$ docker run \
    -d \
    -p 8500:8500 \
    -p 8600:8600/udp \
    --name=badger \
    consul agent -server -ui -node=server-1 -bootstrap-expect=1 -client=0.0.0.0

由于您以detached mode 模式启动容器，-d因此该进程将在后台运行。

您还可以将端口映射设置到本地计算机，并将我们 agents 的 client 接口绑定到 0.0.0.0。

这允许您从本地计算机直接使用 Consul 数据中心，并通过本地主机访问 Consul 的 UI 和 DNS。

最后，您将使用 Docker 的默认桥接网络。

/consul/config请注意，Consul Docker 镜像默认设置 Consul 配置目录。 agents 将加载放置在该目录中的所有配置文件。

配置目录不作为卷公开，并且不会保留数据。Consul 仅在启动期间使用它，并且不存储任何状态。

为了避免安装卷或将文件复制到容器，您还可以通过环境变量将配置 JSON CONSUL_LOCAL_CONFIG 保存到该目录，本教程稍后将介绍这一点。

现在您有了一个小型数据中心，您可以注册服务并执行维护操作。

注册一个服务

在第三个容器中启动服务并向 Consul Client 注册。 基本服务每次被访问时都会增加一个数字并返回该数字。

Pull the container.

$ docker pull hashicorp/counting-service:0.0.2

使用端口转发运行容器，以便您可以通过访问http://localhost:9001从 Web 浏览器访问它。

$ docker run \
   -p 9001:9001 \
   -d \
   --name=weasel \
   hashicorp/counting-service:0.0.2

接下来，您将通过添加在目录中调用的服务定义文件来向 Consul Client 注册 counting 服务 到 consul/config/counting.json。

$ docker exec fox /bin/sh -c "echo '{\"service\": {\"name\": \"counting\", \"tags\": [\"go\"], \"port\": 9001}}' >> /consul/config/counting.json"

由于 Consul Client 不会自动检测配置目录中的更改，因此您需要为同一容器发出重新加载命令。

$ docker exec fox consul reload
Configuration reload triggered

如果您返回启动 Client 的终端窗口，应该有日志条目确认 Consul Client 收到挂断信号、重新加载其配置并 synced counting 服务。

2022-12-15T19:02:24.945Z [INFO]  agent: Synced service: service=counting

停止、启动和重新启动容器

官方 Consul 容器支持停止、启动和重新启动。

要停止容器，请运行docker stop.

$ docker stop <container_id>

要启动容器，请运行docker start.

$ docker start <container_id>

要进行内存中重新加载，请向容器发送 SIGHUP。

$ docker kill --signal=HUP <container_id>

从数据中心删除 Server

只要数据中心中有足够的 Server 来维持quorum，Consul 的Autopilot功能就会处理``删除容器已停止的 Server。

Autopilot 的默认设置已正确配置。

如果您覆盖它们，请确保以下设置合适。

  • cleanup_dead_servers必须设置为 true 以确保从数据中心删除已停止的容器。
  • last_contact_threshold应该相当小，以便快速删除死 Server。
  • server_stabilization_time应足够大（大约几秒），以便不稳定的 Server 在稳定之前不会添加到数据中心。

如果运行当前选举的 Consul Server leader 的容器停止，将会触发 leader 选举。

当使用 重新启动先前停止的 Server 容器，并将其配置为获取新 IP 时，

Autopilot 会将其添加回具有相同 node-id 和 新 IP Address 的docker start <container_id> Raft 对等点集合，之后它可以作为再次 Server 。

备份数据

您可以使用 consul snapshot命令备份 Consul 数据中心。

$ docker exec <container_id> consul snapshot save backup.snap

这会将快照文件保留backup.snap在容器内。如果您不将快照保存到持久卷，那么您将需要使用将 docker cp 快照 移动到容器外部的位置。

$ docker cp <container_id>:backup.snap ./

运行 Consul Enterprise Docker 容器的用户可以运行consul 快照代理来自动保存备份。Consul Enterprise 的快照 agent 还允许您将快照保存到 Amazon S3 和 Azure Blob 存储。

环境变量

Consul Docker 镜像支持通过从 Docker 命令行传入的环境变量进行配置。

Consul agent

CONSUL_LOCAL_CONFIG，CONSUL_CLIENT_INTERFACE并CONSUL_BIND_INTERFACE传递到带有标志的容器-e。

CONSUL_LOCAL_CONFIG支持传递键和值的 JSON 字符串。

要定义datacenter、server并使用 开启调试enable_debug，请在启动 Consul agent 时使用以下代码片段。

$ docker run \
  -d \
  -e CONSUL_LOCAL_CONFIG='{
  "datacenter":"us_west",
  "server":true,
  "enable_debug":true
  }' \
  consul agent -server -bootstrap-expect=3

CONSUL_CLIENT_INTERFACE是一个字符串值，表示 Consul 公开 DNS、gRPC 和 HTTP API 的接口名称。

CONSUL_BIND_INTERFACE是一个字符串值，表示 Consul 用于内部 Consul 集群通信的接口。

-bind在运行时，这些环境变量作为二进制文件的和-client参数传递consul。

常见的实现模式包括对 Client 和绑定参数使用相同的接口。这不是必需的；您可以选择为每个值配置不同的接口。一个例子如下所示。

$ docker run \
  -d \
  -e CONSUL_CLIENT_INTERFACE=en0 \
  -e CONSUL_BIND_INTERFACE=en1 \
  consul agent -server -bootstrap-expect-3

设置CONSUL_ALLOW_PRIVILEGED_PORTS为在 Consul 二进制文件上true运行setcap，允许其绑定到特权端口。

在此示例中，consul agent在端口 53（特权端口）上运行 DNS 服务器，并8.8.8.8通过-recursor参数将上游 DNS 服务器设置为。

$ docker run -d --net=host -e 'CONSUL_ALLOW_PRIVILEGED_PORTS=true' consul agent -dns-port=53 -recursor=8.8.8.8

并非所有 Docker 存储后端都支持此功能（尤其是 AUFS）。请阅读docker-vault 的 github 上的 aufs 问题以获取更多信息。
